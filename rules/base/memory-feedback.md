# Memory: generalize feedback entries

Feedback memories should be written as generalized rules, not tied to the specific session that prompted them. When reviewing an entry that mixes a one-off instance with a generalizable principle, rewrite it around the principle and strip the instance — do not delete the whole entry just because the instance is over.

**Why:** Feedback memories scoped too narrowly to their originating session become disposable once that work lands — the session-specific framing is ephemeral while the underlying rule is durable.

**How to apply:** Lead with the rule in session-agnostic terms; mention the originating instance only as a brief illustration in the **Why**. Before deleting a narrow memory, check whether there is a generalizable kernel worth reframing. Only delete outright when the entry is purely descriptive of code/state that now exists.
